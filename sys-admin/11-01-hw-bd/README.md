## Задание 1. СУБД

Крупная строительная компания, которая также занимается проектированием и девелопментом, решила создать правильную архитектуру для работы с данными. Ниже представлены задачи, которые необходимо решить для каждой предметной области.

Какие типы СУБД, на ваш взгляд, лучше всего подойдут для решения этих задач и почему?   

1.1. Бюджетирование проектов с дальнейшим формированием финансовых аналитических отчётов и прогнозирования рисков. СУБД должна гарантировать целостность и чёткую структуру данных.

1.1.* Хеширование стало занимать длительно время, какое API можно использовать для ускорения работы?

1.2. Под каждый девелоперский проект создаётся отдельный лендинг, и все данные по лидам стекаются в CRM к маркетологам и менеджерам по продажам. Какой тип СУБД лучше использовать для лендингов и для CRM? СУБД должны быть гибкими и быстрыми.

1.2.* Можно ли эту задачу закрыть одной СУБД? И если да, то какой именно СУБД и какой реализацией?

1.3. Отдел контроля качества решил создать базу по корпоративным нормам и правилам, обучающему материалу и так далее, сформированную согласно структуре компании. СУБД должна иметь простую и понятную структуру.

1.3.* Можно ли под эту задачу использовать уже существующую СУБД из задач выше и если да, то как лучше это реализовать?

1.4. Департамент логистики нуждается в решении задач по быстрому формированию маршрутов доставки материалов по объектам и распределению курьеров по маршрутам с доставкой документов. СУБД должна уметь быстро работать со связями.

1.4.* Можно ли к этой СУБД подключить отдел закупок или для них лучше сформировать свою СУБД в связке с СУБД логистов?

1.5.* Можно ли все перечисленные выше задачи решить, используя одну СУБД? Если да, то какую именно?
 
**Ответ**

Решение кейса по выбору СУБД для строительной компании
1.1. Бюджетирование проектов, финансовые отчёты и прогнозирование рисков
Тип СУБД: Реляционная (SQL)
Обоснование: Для финансовых данных критически важны целостность, согласованность и возможность выполнения сложных аналитических запросов с агрегацией. Реляционные СУБД (например, PostgreSQL, Oracle, MS SQL Server) обеспечивают поддержку ACID-транзакций, строгую схему, ссылочную целостность и мощные средства отчётности. Они идеально подходят для бюджетирования и прогнозирования, где данные чётко структурированы.

1.1.* Ускорение хеширования
Решение: Использовать кеширующее API, например Redis или Memcached.
Если хеширование (например, паролей или контрольных сумм) выполняется часто и занимает много времени, можно кешировать результаты вычислений для повторяющихся данных. Также возможно применение аппаратного ускорения или оптимизированных библиотек, но в контексте СУБД наиболее практично внедрить слой кеширования.

1.2. Лендинги и CRM
Для лендингов: Документоориентированная NoSQL (например, MongoDB)
Для CRM: Документоориентированная или реляционная с гибкой схемой (например, PostgreSQL с JSONB)
Обоснование: Лендинги требуют быстрой записи данных с переменной структурой (поля форм могут меняться), что удобно реализовать в MongoDB. CRM содержит данные о клиентах и сделках, где важны связи и структура, но также возможны изменения. PostgreSQL с типом JSONB позволяет сочетать реляционную строгость и гибкость документов.

1.2.* Можно ли закрыть одной СУБД?
**Да, можно использовать PostgreSQL.
Реализация: PostgreSQL поддерживает как классические таблицы для структурированных данных (CRM), так и JSONB для гибких схем (лендинги). Это позволяет унифицировать стек, упростить администрирование и обеспечить транзакционность при необходимости.

1.3. База знаний (нормы, правила, обучающие материалы)
Тип СУБД: Реляционная с простой структурой (например, SQLite или MySQL)
Обоснование: Для хранения иерархически организованных документов (статьи, разделы) достаточно простой реляционной модели с таблицами «категории» и «материалы». Требования к производительности и сложности невысоки, поэтому подойдёт лёгкая СУБД.

1.3.* Использование существующей СУБД
Да, можно использовать любую из ранее выбранных, например PostgreSQL или MongoDB.
Реализация: Создать отдельную базу данных или схему внутри уже работающей СУБД. Например, в PostgreSQL создать схему knowledge_base с таблицами для статей. Это снизит затраты на поддержку новых инстансов.

1.4. Логистика (формирование маршрутов, распределение курьеров)
Тип СУБД: Графовая (например, Neo4j) или реляционная с пространственными расширениями (PostgreSQL + PostGIS)
Обоснование: Задачи маршрутизации требуют эффективной работы со связями (графы дорог, узлы объектов). Графовые СУБД оптимизированы для обхода связей и алгоритмов поиска кратчайших путей. PostGIS также позволяет решать пространственные задачи, но графовые запросы могут быть медленнее.

1.4.* Подключение отдела закупок
Оптимально: Использовать отдельную реляционную СУБД для закупок, интегрированную с логистической через API или шину данных.
Обоснование: Закупки оперируют структурированными данными (поставщики, контракты, цены), для которых лучше подходит реляционная модель. Однако если графовая СУБД поддерживает гибридные данные, можно создать в ней отдельный раздел для закупок. Но проще и надёжнее разграничить системы, обеспечив их взаимодействие.

1.5.* Возможно ли все задачи решить одной СУБД?
**Да, можно использовать PostgreSQL с расширениями.
Обоснование: PostgreSQL покрывает большинство потребностей:

Реляционная модель для финансов и закупок.

JSONB для гибких схем лендингов и CRM.

Рекурсивные запросы (WITH RECURSIVE) для графовых задач.

PostGIS для пространственных данных и маршрутизации.

Высокая надёжность и поддержка ACID.

Хотя специализированные СУБД могут быть эффективнее в узких областях, PostgreSQL обеспечивает универсальность, снижая сложность инфраструктуры и затраты на интеграцию.

## Задание 2. Транзакции

2.1. Пользователь пополняет баланс счёта телефона, распишите пошагово, какие действия должны произойти для того, чтобы транзакция завершилась успешно. Ориентируйтесь на шесть действий.

2.1.* Какие действия должны произойти, если пополнение счёта телефона происходило бы через автоплатёж?

Приведите ответ в свободной форме.
  
**Ответ**

2.1. Транзакция пополнения баланса телефона (ручное пополнение)
Успешное завершение транзакции пополнения счёта мобильного телефона должно гарантировать, что средства списаны с источника (например, банковской карты) и зачислены на счёт абонента, при этом данные остаются согласованными даже в случае сбоя. Ниже представлены шесть ключевых шагов, обеспечивающих атомарность операции:

Начало транзакции (BEGIN TRANSACTION)
Система инициирует транзакцию, фиксируя точку старта. Все последующие операции будут либо выполнены полностью, либо отменены.

Проверка доступности средств на источнике
Выполняется запрос к счёту плательщика (банковская карта, электронный кошелёк) для подтверждения, что сумма пополнения доступна. Это чтение с блокировкой или с учётом изоляции, чтобы избежать двойного списания.

Блокировка (резервирование) суммы на источнике
Средства временно блокируются на счёте плательщика. Например, на карте уменьшается доступный остаток, но фактическое списание ещё не произошло. Это необходимо для предотвращения расходования этих денег в других операциях.

Зачисление средств на счёт телефона
Выполняется запись об увеличении баланса абонента в базе данных оператора связи. Эта операция также должна быть надёжной и учитывать возможные дубликаты (идемпотентность).

Фиксация операции в журнале (лог транзакций)
В системе платёжного шлюза или банка создаётся запись о проведённой операции (идентификатор, сумма, время, статус). Это необходимо для аудита и восстановления после сбоев.

Завершение транзакции (COMMIT)
Если все предыдущие шаги выполнены успешно, транзакция фиксируется. Это означает, что блокировка средств превращается в фактическое списание, а зачисление на счёт телефона становится постоянным. В случае ошибки на любом шаге выполняется откат (ROLLBACK), и все изменения аннулируются.

2.1.* Действия при автоплатеже
Автоплатёж — это регулярное автоматическое пополнение счёта при наступлении определённых условий (например, при снижении баланса ниже порога или в заданную дату). В дополнение к стандартным шагам ручного пополнения добавляются этапы инициирования и проверки условий:

Мониторинг условий автоплатежа
Автоматизированная система (обычно на стороне банка или оператора связи) отслеживает состояние счёта телефона или наступление календарной даты. Как только условие срабатывает (например, баланс упал ниже 100 рублей), генерируется событие для запуска транзакции.

Проверка согласия и лимитов
Перед списанием система проверяет, действительно ли пользователь подписан на автоплатёж, не отозвал ли согласие, и не превышает ли сумма установленных лимитов. Также может выполняться проверка наличия средств на источнике (как в п. 2 основного списка).

Инициирование транзакции (BEGIN TRANSACTION)
Запускается транзакция, как в ручном режиме, но теперь она инициируется системой, а не пользователем.

Выполнение платежа
Далее следуют те же шаги: блокировка суммы на карте, зачисление на счёт телефона, запись в лог.

Уведомление пользователя (опционально)
После успешного коммита система отправляет пользователю push-уведомление, SMS или email о совершённом автоплатеже. Это важно для информирования и предотвращения спорных ситуаций.

Обработка ошибок и повторные попытки
Если транзакция не удалась (например, недостаточно средств), система может сделать несколько повторных попыток через заданные интервалы, а при исчерпании попыток — уведомить пользователя и приостановить автоплатёж до ручного вмешательства.

Таким образом, автоплатёж автоматизирует шаги инициирования и контроля, но внутренняя логика транзакции (атомарность, согласованность) остаётся неизменной.

## Задание 3. SQL vs NoSQL

3.1. Напишите пять преимуществ SQL-систем по отношению к NoSQL.

3.1.* Какие, на ваш взгляд, преимущества у NewSQL систем перед SQL и NoSQL.

Приведите ответ в свободной форме.
  
**Ответ**

3.1. Пять преимуществ SQL-систем перед NoSQL
Строгая схема и целостность данных
Реляционные СУБД требуют чёткого определения схемы (таблицы, типы полей, связи), что обеспечивает согласованность и предотвращает появление "мусорных" данных. Поддержка внешних ключей, ограничений (CHECK, UNIQUE, NOT NULL) и триггеров гарантирует ссылочную целостность, что критично для финансовых, учётных и других систем, где точность данных обязательна.

Поддержка ACID-транзакций
SQL-базы обеспечивают атомарность, согласованность, изоляцию и долговечность (ACID). Это позволяет выполнять сложные операции, состоящие из нескольких шагов, с гарантией, что либо все изменения будут применены, либо ни одного. Для NoSQL (особенно документоориентированных) транзакции часто ограничены одной записью или требуют дополнительных усилий.

Стандартизированный язык запросов (SQL)
SQL является общепринятым стандартом, что упрощает разработку, обучение и переход между различными реляционными СУБД (PostgreSQL, MySQL, Oracle и др.). Богатые возможности SQL позволяют выполнять сложные выборки, объединения, подзапросы, агрегацию и оконные функции, что затруднительно или менее эффективно в NoSQL.

Зрелость и экосистема
Реляционные базы существуют десятилетиями, поэтому они имеют огромное количество инструментов для мониторинга, резервного копирования, миграций, а также обширную документацию и сообщество. Это снижает риски и упрощает сопровождение.

Универсальность и поддержка сложных аналитических запросов
Благодаря строгой структуре и мощному языку запросов SQL-системы хорошо подходят не только для операционной работы (OLTP), но и для аналитики (OLAP) с использованием сложных агрегаций, группировок и соединений. NoSQL чаще специализируется на определённых сценариях (например, быстрое чтение по ключу), и для аналитики может потребоваться отдельный инструмент.

3.1.* Преимущества NewSQL систем перед SQL и NoSQL
NewSQL — это класс современных реляционных СУБД, которые стремятся сочетать гарантии ACID и привычный SQL традиционных баз с горизонтальной масштабируемостью и производительностью NoSQL-решений. Основные преимущества:

Горизонтальное масштабирование без потери ACID
Традиционные SQL-базы сложно масштабировать на кластеры без ручного шардинга, который нарушает транзакционность. NewSQL-системы (например, Google Spanner, CockroachDB, TiDB) изначально проектируются для распределённой архитектуры, автоматически распределяя данные по узлам и поддерживая распределённые транзакции.

Высокая производительность при больших нагрузках
Благодаря современным архитектурным решениям (например, использование протокола Raft для консенсуса, хранение данных в памяти, оптимизация под облачные среды) NewSQL способен обрабатывать миллионы транзакций в секунду, сравнимые с NoSQL, но с гарантиями целостности.

Совместимость с существующими приложениями
Поскольку NewSQL поддерживает стандартный SQL и драйверы JDBC/ODBC, миграция с классических SQL-баз требует минимальных изменений в коде. Это выгодно отличает их от NoSQL, где часто приходится переписывать логику работы с данными.

Автоматическое управление репликацией и отказоустойчивостью
NewSQL-системы встроенно обеспечивают синхронную репликацию и автоматическое восстановление после сбоев узлов, что повышает надёжность без усложнения инфраструктуры.

Поддержка гибридных нагрузок (HTAP)
Некоторые NewSQL-решения (например, TiDB) объединяют транзакционную обработку (OLTP) и аналитику (OLAP) в одной системе, используя отдельные движки для разных типов запросов, что избавляет от необходимости создавать отдельные хранилища данных.

Таким образом, NewSQL предлагает компромисс, сохраняя сильные стороны реляционной модели и добавляя гибкость и масштабируемость, свойственные NoSQL, что делает их идеальным выбором для современных высоконагруженных приложений, требующих строгой согласованности.

## Задание 4. Кластеры

Необходимо производить большое количество вычислений при работе с огромным количеством данных, под эту задачу выделено 1000 машин.

На основе какого критерия будете выбирать тип СУБД и какая модель распределённых вычислений здесь справится лучше всего и почему?

Приведите ответ в свободной форме.

**Ответ**

При выборе типа СУБД и модели распределённых вычислений для кластера из 1000 машин, предназначенного для обработки огромных объёмов данных и выполнения интенсивных вычислений, ключевыми критериями становятся масштабируемость, отказоустойчивость и эффективность параллельной обработки. Важно также учитывать характер данных (структурированные, полуструктурированные, неструктурированные) и типы выполняемых задач (пакетная обработка, потоковая аналитика, итеративные алгоритмы машинного обучения).

Критерии выбора:
Линейная масштабируемость – система должна утилизировать все 1000 машин практически без потери производительности при добавлении узлов.

Отказоустойчивость – автоматическое восстановление после сбоев отдельных узлов без потери данных и остановки вычислений.

Поддержка различных моделей данных – возможность работать как с сырыми данными (файлы, JSON), так и со структурированными таблицами.

Скорость обработки – особенно для итеративных алгоритмов (например, машинное обучение) важна минимизация дискового ввода-вывода.

Богатый набор инструментов – библиотеки для SQL, потоковой обработки, графовых алгоритмов, ML.

Наилучшая модель распределённых вычислений
Наиболее подходящей моделью для такого кластера является парадигма распределённой обработки данных на основе устойчивых распределённых наборов данных (RDD) и направленных ациклических графов (DAG), реализованная в Apache Spark. Почему:

Вычисления в памяти – Spark выполняет промежуточные результаты в оперативной памяти, что критически ускоряет итеративные алгоритмы (например, обучение моделей) по сравнению с дисковыми операциями MapReduce.

Эффективное использование кластера – DAG-оптимизатор строит оптимальный план выполнения, разбивая задачи на стадии и распределяя их по узлам.

Универсальность – Spark поддерживает пакетную обработку (DataFrames, SQL), потоковую (Structured Streaming), графовую (GraphX) и машинное обучение (MLlib) в единой среде.

Отказоустойчивость – через линию происхождения (lineage) RDD Spark может пересчитать утерянные данные без репликации, экономя ресурсы.

Если данные строго структурированы и требуют сложных SQL-запросов с низкой задержкой, можно рассмотреть MPP-СУБД (например, Greenplum, ClickHouse), которые также масштабируются горизонтально и используют массово-параллельную архитектуру. Однако они менее гибки для неструктурированных данных и итеративных вычислений.

Альтернативы и дополнения
Для хранения исходных данных (озеро данных) подойдёт распределённая файловая система (HDFS, S3) или объектное хранилище.

В задачах, требующих реального времени, Spark Streaming можно дополнить очередями сообщений (Kafka).

Таким образом, Apache Spark в связке с распределённым хранилищем (например, HDFS) является оптимальным выбором для кластера из 1000 машин, обеспечивая высокую производительность, отказоустойчивость и универсальность для широкого спектра задач обработки больших данных.
